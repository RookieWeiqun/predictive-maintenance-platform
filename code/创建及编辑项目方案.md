# 创建及编辑项目方案

## 1. 当前前端实现目标

- 新建项目时：先保存项目基础信息，再同步设备、产品、设备检测方案、外围检测方案。
- 编辑项目时：尽量按现有数据增量同步，避免因为重复调用 `ProjectEquipments` 导致重复生成 `inspection_tasks`。
- 设备检测按产品 `productid` 维度判断任务是否已存在。
- 外围检测按 `productid = 0` 维度判断任务数量，并按数量增减。

## 2. 新建项目调用链

### 2.1 保存项目基础信息

调用接口：

- `POST /api/Projects`
- `POST /api/Reports`

说明：

- `POST /api/Projects` 用于创建项目主记录。
- 新建成功后前端会调用 `POST /api/Reports` 初始化一条空报告记录。

### 2.2 同步设备与产品

调用接口：

- 已有设备不调用新增接口。
- 新建设备时调用 `POST /api/Equipments`
- 每台设备对应产品不足时调用 `GET /api/Products/Search?equipmentid=...`
- 如果产品数量不够，再调用 `POST /api/Products`

说明：

- 前端先确保每个设备都有真实 `equipid`。
- 再根据设备数量补齐 `Products`，保证后续设备检测任务能按 `productid` 维度判断。

### 2.3 同步设备检测方案

调用接口：

- `GET /api/ProjectEquipments/ByProject/{projectid}`
- `GET /api/Products/Search?equipmentid=...`
- `GET /api/InspectionTasks/Search?projectid={projectid}&productid={productid}`
- `POST /api/ProjectEquipments`
- `PUT /api/ProjectEquipments`
- `DELETE /api/ProjectEquipments/{peid}`
- `DELETE /api/InspectionTasks/{taskid}`

当前规则：

1. 前端先按型号收集已选设备方案，生成目标的 `equipmentid + templateid` 绑定。
2. 再查询 `GET /api/ProjectEquipments/ByProject/{projectid}`，拿到当前项目已有的设备方案绑定。
3. 对每台设备，再通过 `GET /api/Products/Search?equipmentid=...` 找出这台设备下所有 `productid`。
4. 对每个 `productid`，调用 `GET /api/InspectionTasks/Search?projectid={projectid}&productid={productid}` 检查是否已有设备检测任务。
5. 如果某台设备当前已有同一个模板绑定，则跳过，不再重复创建。
6. 如果某台设备已有绑定，但模板改了，则调用 `PUT /api/ProjectEquipments` 更新该行，由后端重新切换该设备对应任务。
7. 如果某台设备是新增绑定，且它对应的产品任务并非都已存在，则调用 `POST /api/ProjectEquipments` 创建绑定，让后端生成任务。
8. 如果某台设备在编辑时被移除了，则调用 `DELETE /api/ProjectEquipments/{peid}` 删除绑定；同时对该设备下每个 `productid` 搜索任务，并调用 `DELETE /api/InspectionTasks/{taskid}` 逻辑删除旧任务。
9. 如果编辑提交时命中了“同一项目下该设备已存在”，前端会直接跳过这台设备的绑定同步，不提示用户，也不阻塞外围检测继续提交。

关键点：

- 编辑项目时，不再走“先删光全部 `project_equipments` 再重建”的旧逻辑。
- 这样可以避免对于已经存在的 `product inspection_tasks` 再次触发创建。

## 3. 编辑项目加载链

### 3.1 加载项目和设备

调用接口：

- `GET /api/Projects/{projectid}`
- `GET /api/Equipments/ByProject/{projectid}` 或当前前端封装对应接口

说明：

- 用于恢复项目基础信息和设备列表。

### 3.2 加载设备检测方案回填

调用接口：

- `GET /api/ProjectEquipments/ByProject/{projectid}`

说明：

- 前端会把 `project_equipments` 里的 `equipmentid -> templateid` 映射回“设备检测方案（按型号）”的已选值。
- 后续模板候选刷新时，优先保留这个已绑定模板，避免编辑页被默认值覆盖。

### 3.3 加载外围检测方案和数量回填

调用接口：

- `GET /api/InspectionTasks/Search?projectid={projectid}&productid=0`

说明：

- 外围任务固定用 `productid = 0`。
- 返回几条有效任务，就回填成几个电气室。
- 如果没有任务，则回填为“暂时未知”。

## 4. 外围检测同步规则

调用接口：

- `GET /api/InspectionTasks/Search?projectid={projectid}&productid=0`
- `POST /api/ProjectEquipments`
- `PUT /api/InspectionTasks`
- `DELETE /api/InspectionTasks/{taskid}`

规则：

1. 先读取 `GET /api/InspectionTasks/Search?projectid={projectid}&productid=0`。
2. 如果当前电气室数量是“暂时未知”，则删除所有现有外围任务。
3. 如果已有外围任务数量大于目标数量，则对多出的任务调用 `DELETE /api/InspectionTasks/{taskid}`。
4. 如果已有外围任务数量小于目标数量，则按差值次数调用 `POST /api/ProjectEquipments`，每次都传：

```json
{
	"peid": 0,
	"projectid": 123,
	"equipmentid": 0,
	"templateid": 456,
	"ifdel": false
}
```

5. 如果已有外围任务存在但模板换了，则调用 `PUT /api/InspectionTasks` 把现有外围任务的 `templateid` 更新成当前所选外围方案。

## 5. 当前前端文件位置

- 核心提交流程：`frontend/src/pages/project/create/index.vue`
- `ProjectEquipments` API 封装：`frontend/src/api/modules/projectEquipments.ts`
- `InspectionTasks` API 封装：`frontend/src/api/modules/inspectionTasks.ts`

## 6. 当前实现边界

- 设备检测的“避免重复创建”现在已经按现有 `project_equipments` 和现有 `inspection_tasks` 做增量同步。
- 外围检测完全按 `productid = 0` 的任务数控制增减。
- 当前本地后端 `POST /api/InspectionTasks` 只会创建 task 主记录，不会自动生成 taskitems，所以设备补任务仍然优先依赖 `ProjectEquipments` 的新增/更新逻辑生成完整任务。

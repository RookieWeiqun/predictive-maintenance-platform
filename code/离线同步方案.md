# 离线同步方案

## 1. 目标与范围

本文档针对当前系统的“任务执行阶段离线”场景设计，不覆盖后台管理、模板管理、任务创建、项目配置等全量离线能力。

当前边界如下：

1. 任务创建、模板匹配、任务下发：在线完成。
2. iPad 端下载的是“任务包”，不是全量主数据。
3. 工程师到现场后，在无网状态下只做任务执行。
4. 离线阶段允许填写检查结果、拍照、标记跳过/不适用、现场增项。
5. 恢复网络后，将执行结果和图片同步回云端 PostgreSQL 和对象存储。

本方案的目标是：

1. 保证任务执行页面在无网时可完整打开和填写。
2. 允许现场对任务项实例做有限调整，而不是修改模板本身。
3. 保证同步过程可控、可追踪、可审计。
4. 控制第一版复杂度，只实现“任务执行离线最小闭环”。

## 2. 总体设计原则

### 2.1 核心原则

1. 云端 PostgreSQL 是最终事实来源。
2. iPad 本地 SQLite 是任务执行期缓存和待同步存储。
3. 模板只在在线阶段使用，离线阶段只操作任务项实例。
4. 标准任务项不做物理删除，现场“删项”落为业务状态变更。
5. 现场新增项只影响本次任务实例，不直接回写模板。
6. 图片不直接长期存入数据库大字段，数据库只存元数据和路径。

### 2.2 任务包思路

在线端先根据模板生成任务项实例，然后 iPad 下载任务包。任务包中应包含：

1. 任务头信息。
2. 当前任务的任务项实例列表。
3. 每个任务项用于展示和录入的必要字段。
4. 当前已有草稿结果或历史执行结果。
5. 任务执行页面所需的分类路径、排序、展示文案。

离线时不再依赖模板匹配逻辑，不需要再访问模板管理数据。

## 3. 业务语义定义

### 3.1 任务项的两种来源

任务项实例按来源分成两类：

1. `system_generated`
   由模板在线生成的标准任务项。
2. `manual_added`
   工程师在现场手工新增的补充项。

这两个来源必须区分，否则后续无法稳定处理同步、统计和报告。

### 3.2 任务项执行状态

建议增加 `execution_status` 字段，含义如下：

1. `pending`
   待执行。任务包刚下发或尚未填写结果时使用。
2. `completed`
   已执行并已填写结果。
3. `skipped`
   本次现场决定跳过该项，但该项原本是计划中的标准检查项。
4. `not_applicable`
   该项对当前设备或现场不适用。
5. `recheck_required`
   已执行，但需要后续复检或二次确认。

第一版如果不想区分 `skipped` 和 `not_applicable`，也可以先只保留：

1. `pending`
2. `completed`
3. `skipped`
4. `recheck_required`

但推荐最终保留 `not_applicable`，因为它能帮助后续优化模板匹配逻辑。

### 3.3 关于现场“删项”

现场“删项”不建议理解为数据库物理删除，建议改成：

1. 对标准项：改 `execution_status` 为 `skipped` 或 `not_applicable`。
2. 对现场新增项：如果尚未同步，可允许本地删除；如果已同步，建议软删除或标记取消。

这样可以保留原始计划项，避免报表和审计失真。

### 3.4 关于现场“增项”

现场增项应当新增“任务项实例”，而不是修改模板。新增项仅对当前任务生效。

新增项特点：

1. `source_type = manual_added`
2. 不要求回写模板定义。
3. 同步后在云端作为本次任务的补充项保留。
4. 后台如认为有价值，可后续人工纳入模板。

## 4. 云端 PostgreSQL 设计

### 4.1 当前问题

当前 `Taskitem` 模型字段较少，主要适用于基础 CRUD，不足以承接离线执行与同步。当前模型大致包括：

1. `Itemid`
2. `Taskid`
3. `Name`
4. `Categorypath`
5. `Result`
6. `Isnormal`
7. `Isrecheck`
8. `Photopath`
9. `Createtime`

存在的问题：

1. `Itemid` 仅是数据库内自增主键，不适合本地新增后跨端同步。
2. 无法区分标准项和现场新增项。
3. 无法表达待执行、已完成、跳过、不适用、待复检等状态。
4. 单个 `Photopath` 无法承接一项多图。
5. 无版本字段，不利于并发控制和同步冲突判断。

### 4.2 云端任务项实例表建议

建议将云端 `Taskitem` 扩展为“任务项实例表”。第一版最小建议字段如下：

1. `itemid`
   数据库主键，自增，继续保留。
2. `task_item_uuid`
   跨端唯一业务主键，必加。
3. `taskid`
   所属任务。
4. `source_type`
   取值：`system_generated`、`manual_added`。
5. `source_inspection_item_id`
   如果来自模板项则记录模板项 ID，现场新增项为空。
6. `name`
   任务项名称快照。
7. `categorypath`
   分类路径快照。
8. `result`
   当前最终结果值。若第一版只需要单值，继续使用该字段即可。
9. `execution_status`
   任务项执行状态。
10. `isnormal`
    是否正常，可按原逻辑保留。
11. `isrecheck`
    是否复检，可保留，也可逐步由 `execution_status` 接管。
12. `createtime`
    创建时间。
13. `updated_at`
    更新时间，必加。
14. `version`
    版本号，必加。
15. `deleted_at`
    可选，用于软删除现场新增项。

第一版如果想尽量克制，不需要一步到位上太多字段，最少建议加：

1. `task_item_uuid`
2. `source_type`
3. `execution_status`
4. `updated_at`
5. `version`

### 4.3 为什么需要 `task_item_uuid`

`Itemid` 是数据库内部主键，只在云端数据库中有意义。现场离线新增项在同步前并没有云端 `Itemid`，因此需要一个本地和云端都能共享的稳定业务主键。

推荐规则：

1. 云端下发标准项时，同时下发 `task_item_uuid`。
2. iPad 离线新增项时，本地直接生成新的 `task_item_uuid`。
3. 同步到云端时，云端按 `task_item_uuid` 接收并建立记录。

这样可以避免“本地临时 ID 和云端正式 ID”之间反复做映射修补。

### 4.4 为什么需要 `version`

`version` 主要用于乐观并发控制和同步冲突判断。

推荐规则：

1. 云端每条任务项初始 `version = 1`。
2. 每次云端成功更新该记录，`version = version + 1`。
3. iPad 下载任务包时，保存当前 `version`。
4. iPad 回传任务项更新时，同时带上它看到的 `version`。
5. 云端校验：
   - 如果客户端 `version` 和服务端当前版本一致，则允许更新。
   - 如果不一致，则说明该记录在离线期间已被别人改过，应返回冲突。

这本质上就是乐观锁，但在离线同步场景里，它不只是数据库概念，也是同步控制基础。

### 4.5 附件表建议独立

不建议继续依赖单个 `Photopath`。建议新增附件表，例如 `taskitem_attachments`，字段可包括：

1. `attachment_id`
2. `attachment_uuid`
3. `task_item_uuid`
4. `storage_key`
5. `file_name`
6. `mime_type`
7. `size_bytes`
8. `sha256`
9. `created_at`
10. `updated_at`
11. `deleted_at`

这样支持一项多图，也便于对象存储管理。

## 5. iPad 本地 SQLite 设计

### 5.1 总体说明

iPad 本地 SQLite 不是云端数据库副本，而是离线执行期的数据仓储。只需要承接以下能力：

1. 存离线任务包。
2. 保存工程师填写结果。
3. 保存本地图片元数据和文件路径。
4. 保存待同步操作。

建议第一版只做四张表：

1. `offline_task`
2. `offline_task_item`
3. `offline_attachment`
4. `offline_outbox`

### 5.2 `offline_task`

作用：保存当前已下载且可离线执行的任务头信息。

建议字段：

1. `task_uuid`
2. `server_task_id`
3. `task_no`
4. `project_id`
5. `project_name`
6. `equipment_id`
7. `equipment_name`
8. `product_id`
9. `product_name`
10. `status`
11. `downloaded_at`
12. `last_local_edit_at`
13. `sync_status`

### 5.3 `offline_task_item`

作用：保存任务项实例和离线执行结果。

1. `task_item_uuid`
2. `server_item_id`
3. `task_uuid`
4. `source_type`
5. `item_name`
6. `category_path`
7. `result`
8. `execution_status`
9. `is_normal`
10. `is_recheck`
11. `local_updated_at`
12. `sync_status`

本方案当前按最小字段集落地，不再增加 `source_inspection_item_id`、`sort_order`、`dirty_flag`、`is_deleted_local` 等扩展字段。后续如果需要做模板追溯、复杂排序或更细粒度同步控制，再增补即可。

### 5.4 `offline_attachment`

作用：保存图片元数据，图片文件本体放 iPad 本地文件系统。

建议字段：

1. `attachment_uuid`
2. `task_item_uuid`
3. `local_path`
4. `mime_type`
5. `size_bytes`
6. `sha256`
7. `captured_at`
8. `upload_status`
9. `remote_url`

推荐图片存储路径：

`/task-files/{task_uuid}/{task_item_uuid}/{attachment_uuid}.jpg`

### 5.5 `offline_outbox`

作用：保存待同步操作。

建议字段：

1. `op_id`
2. `entity_type`
3. `entity_uuid`
4. `action`
5. `payload_json`
6. `created_at`
7. `retry_count`
8. `status`
9. `last_error`

说明：

1. 每次本地操作后，不要等联网再扫表找差异。
2. 应立即写一条 outbox 记录。
3. 恢复网络后按 outbox 顺序同步。

## 6. 客户端目录建议

本地离线表结构应写在 iPad 客户端工程中的本地数据层，而不是直接写在页面里。建议前端工程分层如下：

```text
src/
   api/
   pages/
   offline/
      db/
         schema.ts
         migrations.ts
         sqlite.ts
         runtime.ts
      repositories/
         taskRepository.ts
         taskItemRepository.ts
         attachmentRepository.ts
         outboxRepository.ts
      services/
         taskPackageService.ts
      index.ts
      types.ts
```

职责说明：

1. `schema.ts`
   定义四张本地表的建表 SQL。
2. `sqlite.ts`
   定义统一的 SQLite 执行接口与初始化入口。
3. `runtime.ts`
   选择原生 Capacitor SQLite 或浏览器调试 fallback。
4. `migrations.ts`
   管理本地表版本和建表 SQL。
5. `repositories/*`
   封装四张本地表的读写。
6. `taskPackageService.ts`
   负责把在线任务包写入本地离线库。
7. `types.ts`
   统一定义离线记录结构和状态枚举。
8. 页面层只调用 repository 和 service，不直接拼 SQL。

## 7. 任务包设计

### 7.1 任务包内容

建议下载接口直接返回单任务执行包，结构包括：

1. 任务头。
2. 任务项实例列表。
3. 当前任务相关的基础展示字段。
4. 已有草稿或已保存结果。
5. 必要的分类路径、排序信息。

不建议只返回 `taskId` 再让 iPad 断网后去拼其他数据。

### 7.2 任务包下载时机

推荐时机：

1. 工程师出发前手动下载。
2. 工程师打开任务详情页时主动点击“下载离线任务包”。
3. 当网络状态较好时自动刷新未开始任务包。

### 7.3 任务包更新规则

建议：

1. 未开始执行前，可重新下载覆盖本地任务包。
2. 已开始执行后，不自动覆盖本地结果。
3. 如任务包云端已变更，提示用户先同步本地草稿，再决定是否刷新。

## 8. 离线执行流程

### 8.1 在线准备阶段

1. 后台在线创建任务。
2. 在线根据模板生成标准任务项实例。
3. 服务端返回单个任务包。
4. iPad 本地存入 `offline_task` 和 `offline_task_item`。

### 8.2 现场离线执行阶段

工程师在无网环境中可执行：

1. 查看任务和任务项。
2. 填写 `result`。
3. 修改 `execution_status`。
4. 标记正常或复检。
5. 拍照并绑定到任务项。
6. 新增补充项。
7. 对标准项标记跳过或不适用。

每次修改都应：

1. 落本地 SQLite。
2. 写入 `offline_outbox`。
3. 更新任务本地修改时间。

### 8.3 恢复网络后的同步阶段

推荐顺序：

1. 先上传附件文件。
2. 再推送任务项变更。
3. 再推送任务状态变更。
4. 服务端返回最新版本、服务端 ID、附件远端地址。
5. 本地更新同步状态并清理成功的 outbox。

## 9. 同步动作设计

建议第一版只支持以下 4 类任务项同步动作：

1. `update_task_item_result`
   更新任务项结果、正常标记、复检标记、执行状态。
2. `skip_task_item`
   将标准项改为 `skipped` 或 `not_applicable`。
3. `add_manual_task_item`
   新增现场补充项。
4. `delete_manual_task_item`
   删除现场新增项。

不建议存在如下动作：

1. `delete_system_generated_task_item`

原因是标准项应保留原始计划语义，不建议被物理删除。

## 10. API 设计建议

当前后端已有常规 CRUD 接口，但离线执行建议新增专门接口，而不是直接复用单条增删改查。

推荐第一版接口：

### 10.1 下载任务包

`GET /api/offline/tasks/{taskId}/package`

返回内容建议包括：

1. 任务头信息。
2. 标准任务项实例。
3. 现场已保存结果。
4. 当前版本号。

### 10.2 同步任务执行结果

`POST /api/offline/tasks/{taskId}/sync`

请求中包括：

1. 任务头更新。
2. 任务项变更列表。
3. 附件元数据。
4. 客户端版本信息。
5. 设备标识。

返回中包括：

1. 成功项列表。
2. 冲突项列表。
3. 云端最新版本。
4. 云端 `itemid` 与 `task_item_uuid` 映射。

### 10.3 上传附件

`POST /api/offline/files/upload`

上传成功后返回：

1. `attachment_uuid`
2. `storage_key`
3. `remote_url`

## 11. 冲突处理建议

虽然当前场景不是全量离线，但仍可能出现冲突。主要是以下情况：

1. 工程师离线执行时，后台把任务关闭或重新指派。
2. 两个设备同时执行同一任务。
3. 离线期间云端有人修改了同一任务项。

建议处理方式：

1. 同步时校验 `version`。
2. 如果版本不一致，服务端返回冲突，不直接覆盖。
3. 前端提示用户该任务已变更，需要人工处理。
4. 对同一任务可增加“领取设备”或“正在执行设备”标识，尽量避免多人同时离线执行。

## 12. 图片策略

### 12.1 本地存储

图片文件本体存 iPad 本地文件系统，不存入 SQLite 大字段。

SQLite 只记录：

1. 本地路径
2. 文件类型
3. 文件大小
4. 哈希值
5. 上传状态

### 12.2 云端存储

推荐：

1. 图片文件上传对象存储。
2. PG 只保存附件元数据。

不建议直接把原图二进制长期存入 PostgreSQL。

### 12.3 任务项与图片绑定

一张图片应明确绑定到某个任务项，而不是只挂在任务头。后续报告回看时，必须知道“这张图对应哪个检查项”。

## 13. 第一版最小落地范围

为了尽快形成可用闭环，建议第一版只做：

1. 在线生成任务项实例。
2. 下载单任务离线包。
3. iPad 本地 SQLite 保存任务和任务项。
4. 离线填写结果。
5. 离线拍照。
6. 现场新增补充项。
7. 对标准项标记跳过。
8. 联网后同步结果和图片。

第一版可以暂缓：

1. 复杂冲突合并界面。
2. 全量主数据离线。
3. 模板本地编辑。
4. 复杂审批流。
5. 自动回写模板优化。

## 14. 推荐实施顺序

### 第一步：云端数据模型补强

优先补：

1. `task_item_uuid`
2. `source_type`
3. `execution_status`
4. `updated_at`
5. `version`

并新增附件表。

### 第二步：下载任务包接口

先保证 iPad 可以下载单任务的完整执行包。

### 第三步：本地 SQLite 落地

先完成：

1. `offline_task`
2. `offline_task_item`
3. `offline_attachment`
4. `offline_outbox`

### 第四步：离线执行页面接本地仓储

任务执行页在离线时从 SQLite 读取和写入，不再依赖在线接口。

### 第五步：同步接口和附件上传

先实现单任务同步闭环，再考虑批量任务同步。

## 15. 最终结论

本项目当前最适合的不是“全量离线数据库方案”，而是“任务执行离线包方案”。

总结如下：

1. 在线端先根据模板生成任务项实例。
2. iPad 下载任务包，只缓存任务执行需要的数据。
3. 离线时只改任务项实例结果，不做模板管理。
4. 标准项不做物理删除，现场删项改为状态变更。
5. 现场增项作为本次任务的补充项保留。
6. 本地 SQLite 负责离线执行和待同步数据。
7. 云端 PostgreSQL 负责最终事实、审计和历史留存。
8. 图片走本地文件系统加对象存储，不长期塞进数据库大字段。

这套方案可以在保持当前系统结构基本不变的前提下，较低风险地支持 iPad 现场离线执行。

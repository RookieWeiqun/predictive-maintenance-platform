# 离线同步接口草案

本文档给后端同事说明当前平板离线端建议提交的同步数据格式。

补充说明：当前前端“下载任务”还没有调用专门的任务包接口，而是前端自己串以下接口拼包：

1. `GET /api/InspectionTasks/{id}`
2. `GET /api/InspectionTemplates/{templateid}`
3. `GET /api/Taskitems/ByTask/{taskid}`
4. `GET /api/Products/Search?productid=...`
5. `InspectionCategories + InspectionItems` 两张表接口后前端再按 `templateid` 过滤拼树

这条链可以工作，但接口分散、调用较长，而且任务执行快照仍然依赖模板表结构。建议后端新增“任务包下载接口”，让前端直接下载可离线执行的单任务快照。

## 1. 设计原则

1. 云端 PostgreSQL 是主数据源。
2. 平板离线端只提交本地发生变化的增量数据，不提交整库快照。
3. 本地 `offline_outbox` 是前端重试机制，不建议作为后端接口契约直接暴露。
4. 后端接口建议按“任务维度”接收同步数据，一个任务内带多个已变更任务项。

## 2. 推荐接口

推荐先预留一个批量同步接口：

`POST /api/offline-sync/tasks/batch`

请求体为一个批次，里面包含 0..N 个发生本地变更的任务。

## 2.1 推荐新增下载接口

建议新增单任务包下载接口：

`GET /api/offline/tasks/{taskId}/package`

### 设计目标

1. 返回一个前端可直接落本地的完整执行包。
2. 下载后平板离线执行不再依赖 `inspection_template / inspection_category / inspection_item / product` 表查询。
3. 任务项实例自带渲染快照，避免前端再按模板二次拼装。

### 任务包字段说明

#### A. `task`

1. `task_id`
   云端 `InspectionTask.taskid`。
2. `task_uuid`
   前端本地任务唯一键。若当前阶段不单独引入 UUID，可先等于 `task_id` 的字符串形式。
3. `task_no`
   展示用任务编号。
4. `project_id`
   项目 ID。
5. `project_name`
   项目名称快照，避免离线再查项目表。
6. `template_id`
   模板 ID。
7. `template_name`
   模板名称快照。
8. `inspection_type`
   模板检测类型，设备检测/外围检测。
9. `status`
   云端任务状态。
10. `assigned_user_id`
    可选，任务指派人。
11. `download_version`
    当前任务包版本号，用于后续刷新任务包时判断是否覆盖本地。
12. `product_snapshot`
    当前任务对应产品快照。这里不要求单独建数据库列，下载任务包时可通过 `inspection_tasks.productid -> products` 现查现拼。

#### B. `product_snapshot`

1. `product_id`
   产品 ID。
2. `mlfb`
   产品型号。
3. `serialno`
   产品序列号。

说明：

1. `product_snapshot` 建议作为接口返回结构，而不是强制数据库新增字段。
2. 平板离线执行只需要这几个产品快照字段，不需要下载整张 `products` 表。

#### C. `task_items[]`

1. `task_item_uuid`
   任务项实例唯一键。
2. `server_item_id`
   云端任务项主键。你们现在 `Taskitem.Itemid` 已是 `Guid`，如果后端确定它可稳定作为跨端业务标识，那么这里可直接用它。
3. `source_type`
   `system_generated` 或 `manual_added`。
4. `source_inspection_item_id`
   可选，若来自模板项则保留原始模板项 ID，便于追溯。
5. `item_name`
   本次任务项名称快照。
6. `category_path`
   本次任务的展示分类路径快照，例如 `环境检查 / 柜体环境`。
7. `execution_status`
   当前任务项执行状态。
8. `is_normal`
   是否正常。
9. `is_recheck`
   是否复检。
10. `version`
    任务项版本号，用于同步时乐观并发控制。
11. `updated_at`
    最近更新时间。
12. `render_schema_json`
    任务项渲染快照，建议真正落到任务项实例层。
13. `result_snapshot`
    当前已有结果快照。若任务尚未执行，可为空值结构。

#### D. `render_schema_json`

推荐至少包含：

1. `value_type`
2. `rule_type`
3. `threshold`
4. `sort_order`
5. `priority`

后续如有需要，可继续扩展：

1. `unit`
2. `normal_values`
3. `operation_guide`
4. `placeholder`
5. `default_value`

### 为什么建议新增这个接口

1. 当前前端下载任务要调用多组接口后本地拼包，链路长且脆弱。
2. 当前模板结构仍然依赖 `inspection_category + inspection_item` 二次拼装，不利于真正离线执行。
3. 新接口返回任务实例 + 渲染快照后，前端下载即落地，离线执行不再依赖模板表。
4. 后续若支持任务包刷新、任务包版本控制，也更容易统一处理。

## 4. 字段说明

### 4.1 批次头

1. `protocol_version`
   同步协议版本，便于后续接口升级。
2. `generated_at`
   前端生成本次同步批次的时间。
3. `sync_mode`
   当前建议固定为 `delta`，表示仅同步发生变更的数据。
4. `source.app`
   客户端标识。
5. `source.platform`
   运行平台，当前一般为 `android`。

### 4.2 任务头

1. `task_uuid`
   前端本地任务主键，目前与 `server_task_id` 保持一致。
2. `server_task_id`
   云端 `InspectionTask.taskid`。
3. `task_no`
   任务编号展示字段。
4. `project_id`
   云端项目 ID。
5. `scheme_id`
   模板 ID。
6. `status`
   本地任务状态，例如 `pending`、`in-progress`、`completed`。
7. `local_updated_at`
   该任务最近一次本地改动时间。

### 4.3 任务项

1. `task_item_uuid`
   前端本地任务项唯一键。
2. `server_item_id`
   云端 `Taskitem.itemid`，若为本地新增项则可能为空。
3. `source_type`
   `system_generated` 或 `manual_added`。
4. `execution_status`
   `pending`、`completed`、`skipped`、`not_applicable`、`recheck_required`。
5. `is_normal`
   当前检查结果是否正常。
6. `is_recheck`
   是否需要复检。
7. `result.raw`
   前端原始 JSON 字符串，便于问题排查。
8. `result.result`
   归一化后的结果枚举，当前主要是 `normal`、`warning`、`abnormal`。
9. `result.remarks`
   备注文本。
10. `result.data_fields`
    检查项下的结构化录入值，键是前端字段 ID，值是录入内容。

### 4.4 附件

1. `attachment_uuid`
   本地附件唯一键。
2. `task_item_uuid`
   附件所属任务项。
3. `local_path`
   当前是客户端本地文件路径，仅用于客户端内部识别。
4. `mime_type`
   文件 MIME 类型。
5. `size_bytes`
   文件大小。

注意：如果后端要真正接收图片，建议改成两段式同步：

1. 先上传文件到对象存储，返回 `object_key`。
2. 再在任务同步接口里提交 `object_key`，不要直接依赖 `local_path`。

## 6. 前后端约定建议

1. 若 `server_item_id` 为空且 `source_type=manual_added`，后端应创建新的任务项记录。
2. 若 `server_item_id` 有值，后端按更新处理。
3. 后端成功接收后，前端会把对应本地记录标记为 `synced`。
4. 如果某个任务项失败，建议按任务项粒度返回失败原因，避免整任务整批次回滚。
5. 若后端已经根据业务规则禁止某种变更，应明确返回该任务项失败原因，前端会保留 `failed` 状态等待人工处理。

# 离线同步接口草案

本文档说明当前平板离线能力对应的真实接口链路、数据格式和已知边界，供前后端对齐。

当前前端已经实现以下闭环：

1. 在线任务列表按任务下载。
2. 下载后本地离线执行任务项。
3. 每个任务项最多拍 3 张照片并保存在本地私有目录。
4. 离线任务列表支持勾选多个任务上传。
5. 上传时严格按“先图片，后任务明细”的顺序逐任务串行执行。

## 1. 设计原则

1. 云端 PostgreSQL 是主数据源，本地仅保存离线执行所需快照和待同步增量。
2. 同步粒度按任务处理，不做整库同步。
3. 前端本地 `offline_outbox` 仅用于重试控制，不作为后端接口契约暴露。
4. 当前上传采用真实业务接口，不再使用单独的离线批量同步接口。

## 2. 当前真实链路

### 2.1 下载

前端在在线任务列表中点击“下载任务”后，按任务调用：

`GET /api/InspectionTasks/{id}/detail`

当前下载不是调用新的“任务包接口”，而是直接使用这个详情接口作为离线任务包来源。

前端下载后会完成以下动作：

1. 将任务头写入本地 `offline_task`。
2. 将任务项写入本地 `offline_task_item`。
3. 将任务项渲染所需结构转换后缓存到本地模板缓存，供离线执行页直接渲染。
4. 不会修改 backend 数据。

### 2.2 执行

离线执行页直接读取本地离线表：

1. `offline_task`
2. `offline_task_item`
3. `offline_attachment`
4. `offline_outbox`

执行过程中：

1. 用户填写任务项结果、备注、复检标记。
2. 用户可为每个任务项拍照或选择照片。
3. 每个任务项照片数量当前限制为最多 3 张。
4. 照片保存到应用私有目录，并在本地附件表中记录相对路径。
5. 任务项的显示条件、操作指导、建议规则、建议内容、隐患内容、维护说明来自下载时缓存的本地模板结构。

### 2.3 上传

离线任务列表支持勾选多个任务上传，上传时按任务串行处理：

1. 先上传当前任务下所有待同步图片。
2. 当前任务所有图片上传成功后，再上传任务明细。
3. 成功后将任务、任务项、附件和 outbox 状态统一标记为 `synced`。
4. 某个任务失败时，不回滚前面已成功任务。

`GET /api/InspectionTasks/{id}/detail`

说明：

1. 前端兼容 `taskitems` 和 `task_items` 两种字段名。
2. 前端兼容驼峰和下划线字段名，例如 `downloadedAt` 与 `downloaded_at`。
3. `executionStatus` 在下载后会被前端归一化为本地状态字符串。
4. 任务项扩展元数据当前兼容两种来源：
   - 放在 `renderSchemaJson` 内部，例如 `displayCondition`、`suggestionContent`
   - 直接平铺在 `taskitems[]` 顶层，例如 `recommendedrules`、`recommendationcontent`、`hiddenhazardcontent`、`maintenanceinstructions`、`displaycondition`
5. 当前前端下载时会把这两部分数据合并后再写入离线侧，避免只解析到一半。

### 3.3 下载后本地落库存储字段

任务头当前会落到本地的关键字段包括：

1. `task_uuid`
2. `server_task_id`
3. `task_no`
4. `serial_no`
5. `assigned_user_name`
6. `assigned_user_id`
7. `download_device_name`
8. `project_id`
9. `project_name`
10. `scheme_id`
11. `scheme_name`
12. `product_id`
13. `inspection_type`
14. `version`
15. `device_model`
16. `downloaded_at`
17. `local_updated_at`

说明：

1. `local_updated_at` 当前按最近一次成功上传时间使用，不再表示本地编辑时间。

任务项当前会落到本地的关键字段包括：

1. `task_item_uuid`
2. `server_item_id`
3. `task_uuid`
4. `source_type`
5. `item_name`
6. `category_path`
7. `result`
8. `display_condition`
9. `operation_guide`
10. `recommended_rules`
11. `recommendation_content`
12. `hidden_hazard_content`
13. `maintenance_instructions`
14. `execution_status`
15. `is_normal`
16. `is_recheck`
17. `local_updated_at`
18. `sync_status`

说明：

1. 任务项 `local_updated_at` 当前按最近一次成功上传时间使用，上传时作为后端 `updatetime` 回传。
2. 检测项渲染和执行所需的扩展元数据不会拆列写入 `offline_task_item`，而是跟任务下载时的模板结构一起缓存到本地 `task_scheme_{task_uuid}_{scheme_id}` 中。
3. 这批本地模板缓存字段当前包括：
   - `displayCondition`
   - `operationGuide`
   - `suggestionRule`
   - `suggestionContent`
   - `hazardContent`
   - `maintenanceDescription`
4. 离线执行页渲染任务项、自动带出建议内容、显示条件判断等能力，读取的就是这份本地模板缓存，而不是 `offline_task_item` 的独立列。

## 4. 本地执行与附件存储

### 4.1 本地表

当前前端离线侧使用以下本地表：

1. `offline_task`
2. `offline_task_item`
3. `offline_attachment`
4. `offline_outbox`

### 4.2 本地照片保存

图片实际保存在应用私有目录，例如：

```text
task-photos/{task_uuid}/{task_item_uuid}/{timestamp}-{attachment_uuid}.jpg
```

`offline_attachment.local_path` 保存该相对路径，仅供客户端本地读取文件时使用，不作为服务端持久化路径。

### 4.3 当前已知边界

服务端详情接口虽然已经返回 `taskitems[].attachments`，但当前前端只会把本地新拍照片写入 `offline_attachment`。

也就是说：

1. 已经存在于服务端的历史附件，目前不会在下载时自动拉取文件到本地。
2. 检测项扩展元数据当前会以独立列写入 `offline_task_item`，字段名与后端语义保持一一对应。
3. 当前对应关系如下：
  - `display_condition` <-> `displaycondition`
  - `operation_guide` <-> `operationguide`
  - `recommended_rules` <-> `recommendedrules`
  - `recommendation_content` <-> `recommendationcontent`
  - `hidden_hazard_content` <-> `hiddenhazardcontent`
  - `maintenance_instructions` <-> `maintenanceinstructions`
4. 本地模板缓存 `task_scheme_{task_uuid}_{scheme_id}` 仍然保留，用于离线执行页恢复树结构和字段渲染；但上述扩展元数据不再只依赖缓存，离线页也可直接从 `offline_task_item` 独立列读取。
`PUT /api/Products`

请求类型：`application/json`

当前离线上传阶段，设备相关字段不再通过任务接口回传，而是通过产品接口更新。

当前前端实际关注字段包括：

1. `productid`
2. `equipid`
3. `mlfb`
4. `serialno`
5. `equipmentname`
6. `equipmentnumber`
7. `department`

当前请求体示例：

```json
{
  "productid": 14,
  "equipid": 9,
  "mlfb": "6SE70-7HG84-1JA1",
  "serialno": "SN-2026-001",
  "equipmentname": "主传动柜",
  "equipmentnumber": "EQ-01",
  "department": "轧钢一车间"
}
```

说明：

1. `serialno` 不再通过任务接口上传，而是跟 `equipmentname`、`equipmentnumber`、`department` 一起通过 `Products` 接口更新。
2. 当前前端会先根据任务头里的 `productid` 调用 `GET /api/Products/{id}` 读取现有产品信息，再用本地值覆盖后提交 `PUT /api/Products`。
3. 如果本地当前只修改了 `serialno`，其余设备字段会沿用服务端现有值回传，不会被前端置空。

### 5.2 图片上传接口

`POST /api/Attachments/upload`

请求类型：`multipart/form-data`

表单字段：

1. `ItemId`：任务项 ID，UUID 字符串。
2. `Files`：图片文件数组。

说明：

1. 前端按任务项分组上传图片。
2. 一个任务项最多上传 3 张图。
3. 当前前端实现为先上传任务内所有待同步图片，再提交任务明细。

### 5.3 任务明细上传接口

`PUT /api/InspectionTasks/{id}/detail`

其中 `{id}` 为服务端任务主键 `taskid`。

5. `serialno` 已从任务明细上传体中移除，不再通过该接口更新。

## 6. 字段约定

### 6.1 任务头

1. `taskid`：服务端任务主键，同时也是 `PUT /api/InspectionTasks/{id}/detail` 中的 `{id}`。
2. `projectid`、`templateid`、`productid`、`inspectiontype`：按下载详情中的任务头字段回传。
3. `status`：任务状态，前端本地统一归一化后再映射回后端所需数值。
4. `version`：任务级版本号。
5. `downloadedAt`、`localUpdatedAt`、`downloadDeviceName`、`assignedusername`：前端本地保留并原样回传的追溯字段。
6. `serialno` 虽然仍会保存在本地任务头中供页面展示和离线编辑使用，但上传时不再通过任务接口回传。

### 6.2 任务项

1. `itemid`：任务项主键 UUID。
2. `executionStatus`：当前前端映射为 `1=pending`、`2=completed`、`3=skipped`、`4=not_applicable`、`5=recheck_required`。
3. `sourceType`：当前前端映射为 `1=system_generated`、`2=manual_added`。
4. `taskresult`：字符串，内部是 JSON。
5. `inspectionitemid`：若任务项来自模板项，则回传原始模板项 ID。

`taskresult` 内部 JSON 当前前端主要使用如下结构：

```json
{
  "value": "26.5",
  "recheckvalue": null,
  "remarks": "",
  "resultState": "normal"
}
```

### 6.3 附件

1. 图片文件本体通过 `POST /api/Attachments/upload` 上传。
2. 前端从本地文件系统读取 `offline_attachment.local_path` 指向的文件，再作为 `Files` 发送。
3. 任务明细接口当前不再单独提交附件列表。

### 6.4 产品设备字段

1. `serialno`、`equipmentname`、`equipmentnumber`、`department` 统一视为产品维度字段。
2. 这些字段在离线上传时通过 `PUT /api/Products` 更新，而不是通过任务接口更新。
3. 报告页中的设备清单与问题块优先读取 `GET /api/Products/{id}` 或 `GET /api/Products/Search` 返回的真实字段。

## 7. 推荐同步顺序

前端勾选离线任务列表中的多个任务后，当前实际执行顺序如下：

1. 读取勾选任务集合。
2. 按任务逐个串行处理，不并发提交多个任务。
3. 先根据当前任务的 `productid` 调用 `GET /api/Products/{id}` 读取产品当前信息。
4. 再调用 `PUT /api/Products` 更新设备相关字段，包括 `serialno`、`equipmentname`、`equipmentnumber`、`department`。
5. 对当前任务读取全部待上传图片。
6. 按任务项调用 `POST /api/Attachments/upload` 上传图片。
7. 当前任务图片全部成功后，调用 `PUT /api/InspectionTasks/{id}/detail` 上传任务执行明细。
8. 服务端返回成功后，前端把当前任务、任务项、附件和 outbox 状态统一标记为 `synced`。

## 8. 结论

当前前端侧“下载 -> 离线执行 -> 上传”主链路已经成立，且无需修改 backend 即可工作。

当前唯一明确未闭合的点是：

1. 服务端历史附件虽然已在详情接口返回，但前端尚未在下载阶段把这些附件文件拉到本地并纳入离线展示。

如果后续要补这一点，建议新增前端侧能力：

1. 根据详情接口返回的附件信息下载远端文件。
2. 将文件保存到应用私有目录。
3. 同步写入 `offline_attachment`，使离线执行页能直接展示历史图片。

# 在线数据上传接口方案

## 1. 目标

本文档定义平板离线执行后的在线上传方案，覆盖以下范围：

1. 按任务上传，不做整库同步。
2. 每个任务内同时上传任务头、任务项结果和现场照片。
3. 现场照片每个任务项最多 3 张，本地先保存到平板私有目录。
4. 恢复网络后，由离线任务列表勾选多个任务，按任务逐个上传。

## 2. 上传原则

### 2.1 任务粒度

上传粒度与下载任务包保持一致，采用“一个任务一个任务上传”的策略。

原因：

1. 失败隔离简单，某个任务失败不会阻塞其它任务。
2. 前端离线列表可以直接勾选任务并逐个重试。
3. 现场照片通常跟任务项强绑定，任务级事务更容易追踪。

### 2.2 图片先传，任务明细后传

当前按真实接口采用两步上传：

1. 先调用 `POST /api/Attachments/upload` 上传当前任务内待同步图片。
2. 图片全部上传成功后，再调用 `PUT /api/InspectionTasks/{id}/detail` 上传任务与任务项数据。

不在任务 JSON 中传图片二进制或 Base64。

### 2.3 失败处理

1. 如果图片上传失败，则当前任务不进入任务数据提交阶段。
2. 如果图片都成功但任务 JSON 提交失败，则保留图片上传结果和任务待上传状态，等待重试提交。
3. 后端需要支持幂等，避免前端重试时重复创建任务项附件。

## 3. 实际接口

### 3.1 图片上传接口

`POST /api/Attachments/upload`

请求类型：`multipart/form-data`

表单字段：

1. `ItemId`：任务项 ID，类型为 UUID 字符串。
2. `Files`：图片文件，可多张。

说明：

1. 一个任务项最多 3 张图。
2. 前端按任务项分组上传图片，同一个任务项的待传图片放在同一次请求里。
3. 当前前端实现是先上传任务内全部待同步图片，再提交任务明细。

### 3.2 任务明细上传接口

`PUT /api/InspectionTasks/{id}/detail`

其中 `{id}` 为服务端任务主键 `taskid`。

## 4. 数据格式

### 4.1 前端本地数据来源

当前前端离线端已经有以下本地表：

1. `offline_task`
2. `offline_task_item`
3. `offline_attachment`
4. `offline_outbox`

其中图片文件实际保存在应用私有目录，例如：

```text
task-photos/{task_uuid}/{task_item_uuid}/{timestamp}-{attachment_uuid}.jpg
```

`offline_attachment.local_path` 保存的是这个本地相对路径，仅供客户端读取本地文件，不应直接作为服务端持久化路径。

### 4.2 任务明细上传请求体

请求体结构：

```json
{
	"task": {
		"taskid": 68,
		"projectid": 8,
		"templateid": 21,
		"status": 1,
		"taskNo": "EM64-XW9F-57",
		"assigneduserid": null,
		"productid": 14,
		"inspectiontype": 1,
		"ifdel": false,
		"version": 7,
		"downloadedAt": "2026-04-24T01:00:00",
		"localUpdatedAt": "2026-04-24T08:28:12.000Z",
		"downloadDeviceName": "设备1",
		"serialno": "SN-2026-001",
		"assignedusername": "张三"
	},
	"taskitems": [
		{
			"itemid": "016d9b85-51fc-415e-ab56-25d5f89695e6",
			"taskid": 68,
			"taskname": "PER板是否正常，无腐蚀",
			"categorypath": "整流单元/元件性能检测/CUR、PER检测",
			"taskresult": "{\"value\":\"是\",\"recheckvalue\":null,\"remarks\":\"\",\"resultState\":\"normal\"}",
			"isnormal": true,
			"isrecheck": false,
			"photopath": null,
			"executionStatus": 2,
			"updatetime": "2026-04-24T08:12:10.000Z",
			"version": 1,
			"sourceType": 1,
			"inspectionitemid": 349
		}
	]
}
```

说明：

1. `ifdel` 当前固定传 `false`。
2. 当前前端会把任务项图片先上传，`detail` 接口里 `photopath` 先传 `null`。
3. 当前前端会回传任务级 `version`，任务项 `version` 使用服务端详情中的当前值回传。

## 5. 字段说明

### 5.1 任务头

1. `taskid`：服务端任务主键，同时也是 `PUT /api/InspectionTasks/{id}/detail` 中的 `{id}`。
2. `projectid`、`templateid`、`productid`、`inspectiontype`：按服务端任务字段原样回传。
3. `status`：任务状态。当前前端映射为 `1=进行中`、`2=完成`、`3=未开始`。
4. `ifdel`：固定传 `false`。
5. `version`：任务级版本号。
6. `downloadedAt`、`localUpdatedAt`、`downloadDeviceName`、`serialno`、`assignedusername`：前端本地补充信息，用于追溯现场执行来源。

### 5.2 任务项

1. `itemid`：任务项 UUID。
2. `executionStatus`：当前前端映射为 `1=pending`、`2=completed`、`3=skipped`、`4=not_applicable`、`5=recheck_required`。
3. `sourceType`：`1=system_generated`，`2=manual_added`。
4. `isrecheck`：每个任务项都传，默认 `false`。
5. `taskresult`：字符串，内部仍是 JSON，对应下面结构。

```json
{
	"value": "26.5",
	"recheckvalue": null,
	"remarks": "",
	"resultState": "normal"
}
```

6. 如果某类任务项需要用户补录复检值，则 `recheckvalue` 放在 `taskresult` 里一起上传。

### 5.3 图片

1. 图片文件本体通过 `POST /api/Attachments/upload` 上传。
2. 前端本地只保存 `offline_attachment.local_path`，上传时从本地文件系统读出后作为 `Files` 发送。
3. 任务明细接口当前不再单独提交附件列表。

## 6. 推荐上传顺序

前端勾选离线任务列表中的多个任务后，按如下流程执行：

1. 读取勾选任务集合。
2. 按任务循环处理，不并发提交多个任务。
3. 对每个任务先读取该任务所有待上传图片。
4. 按任务项调用 `POST /api/Attachments/upload` 上传图片。
5. 图片全部成功后，调用 `PUT /api/InspectionTasks/{id}/detail`。
7. 服务端返回成功后，前端把当前任务、任务项、附件本地状态统一标记为 `synced`。

## 7. 离线任务列表勾选要求

前端离线任务列表应支持勾选上传，建议规则如下：

1. 仅允许勾选存在待上传变更的任务。
2. 点击“上传数据”时，只上传当前勾选任务。
3. 如果某任务无待上传变更，对应勾选框置灰。
4. 支持表头“全选当前筛选结果中的待上传任务”。

## 8. 当前前端实现说明

1. 离线任务列表支持勾选多个任务上传。
2. 仅允许勾选存在待同步变更的任务。
3. 上传时严格按任务串行处理。
4. 每个任务内严格按“先图片，后任务明细”执行。
5. 任一任务上传失败时，当前任务停止，已成功的前序任务不回滚。

# 模板方案

## 1. 目标

模板的最终目标不是只把 Excel 导进去，而是让前端在“执行任务”页面可以根据以下三个字段动态渲染表单：

- `value_type`
- `rule_type`
- `rule`（对应数据库 `inspection_items.threshold`）

这三个字段的职责建议固定为：

- `value_type`：决定前端渲染什么输入控件
- `rule_type`：决定采用什么校验/判定规则
- `rule`：该规则的具体参数，统一存 JSON 字符串

---

## 2. 关于 `{"normal_value": "是"}` 能不能表达含义

可以表达，但前提是语义要定清楚。

如果：

- `value_type = boolean`
- `rule_type = boolean_equal`
- `rule = {"normal_value": "是"}`

那么它表达的含义就是：

> 该检测项的可选值是“是 / 否”，其中“是”表示正常。

也就是说：

- 用户在表单里选择“是” -> 判定正常
- 用户在表单里选择“否” -> 判定异常或非正常

这个结构本身是能表达业务含义的。

但要注意：

- 如果 `value_type = boolean`，前端必须约定布尔类默认渲染为“是 / 否”单选或切换控件
- 如果未来不仅有“是 / 否”，而是“开 / 关”、“有 / 无”、“正常 / 异常”，那就不能只靠 `normal_value`，还需要额外的显示选项定义

所以推荐：

- 简化版：`boolean_equal + normal_value`
- 扩展版：`boolean_equal + options + normal_value`

例如：

```json
{
	"options": ["是", "否"],
	"normal_value": "是"
}
```

这样前端渲染和结果判定都更明确。

---

## 3. 推荐的动态渲染模型

建议按下面的原则设计：

- 一个检测项只录入一个值
- 一个检测项只对应一个 `value_type`
- 一个检测项只对应一个 `rule_type`
- `rule` 统一保存为 JSON 字符串

执行页动态渲染逻辑：

1. 先根据 `value_type` 渲染控件
2. 再根据 `rule_type + rule` 渲染提示、校验和自动判定逻辑

---

## 4. 推荐支持的 `value_type`

### 4.1 `number`

含义：数值输入。

前端渲染：数字输入框。

示例：

- 温度
- 湿度
- 电流
- 电压

### 4.2 `boolean`

含义：二选一判断。

前端渲染：单选按钮 / 切换开关 / 是否按钮组。

示例：

- 是否凝露
- 是否异响
- 是否松动

### 4.3 `enum`

含义：枚举值选择。

前端渲染：下拉框 / 单选列表。

示例：

- 颜色
- 外观等级
- 污染程度

### 4.4 `text`

含义：文本输入。

前端渲染：文本框。

示例：

- 铭牌信息
- 备注型录入
- 异常描述

### 4.5 `date`

含义：日期输入。

前端渲染：日期选择器。

示例：

- 保养日期
- 校验日期

如果当前版本先不做日期类，可以暂时不开放，但规范里建议预留。

---

## 5. 第一版支持的 `rule_type` 与 `rule` 格式

第一版先只支持下面 4 种组合：

1. `number + number_range`
2. `boolean + boolean_equal`
3. `enum + enum`
4. `text + text_pattern`

其他组合先不做，避免前后端规则定义越来越散。

### 5.1 `number_range`

适用：`value_type = number`

含义：数值必须落在某个范围内。

推荐 `rule`：

```json
{
	"min": -5,
	"max": 20,
	"unit": "℃",
	"precision": 1,
	"step": 0.1
}
```

字段说明：

- `min`：最小值
- `max`：最大值
- `unit`：显示单位，可选
- `precision`：小数精度，可选
- `step`：输入步长，可选

前端渲染行为：

- 渲染数字输入框
- 显示范围提示，例如：`范围 -5 ~ 20 ℃`
- 如果超范围，可标记异常或提示用户

### 5.2 `boolean_equal`

适用：`value_type = boolean`

含义：布尔/二选一值中，某个值表示正常。

推荐 `rule`：

```json
{
	"options": ["是", "否"],
	"normal_value": "是"
}
```

最小可用版也可以写成：

```json
{
	"normal_value": "是"
}
```

但更推荐带 `options`，因为这样前端不需要硬编码“是否”。

前端渲染行为：

- 渲染两个选项，例如“是 / 否”
- 如果用户选中 `normal_value`，则该项可自动判定为正常

### 5.3 `enum`

适用：`value_type = enum`

含义：值从固定枚举列表中选择。

推荐 `rule`：

```json
{
	"options": ["纯白", "米白", "乳白"],
	"normal_values": ["纯白", "米白"]
}
```

字段说明：

- `options`：前端可供选择的枚举项
- `normal_values`：哪些值视为正常，可选

如果不需要自动判定正常/异常，也可以简化为：

```json
{
	"options": ["纯白", "米白", "乳白"]
}
```

前端渲染行为：

- 渲染下拉框或单选组
- 如果配置了 `normal_values`，则可自动计算是否正常

### 5.4 `text_pattern`

适用：`value_type = text`

含义：文本需要满足某个格式。

推荐 `rule`：

```json
{
	"pattern": "^[A-Z0-9-]+$",
	"message": "仅允许大写字母、数字和连字符"
}
```

也允许 `rule` 为空。
··
当 `rule` 为空时，语义定义为：

- 渲染普通文本输入框
- 不做正则校验
- 只采集文本值本身

也就是说：

- `value_type = text`
- `rule_type = text_pattern`
- `rule = null` 或空字符串

可以理解成“文本输入，但当前没有额外格式限制”。

前端渲染行为：

- 渲染文本框
- 如果 `rule.pattern` 存在，则按正则校验
- 如果校验失败，则提示 `message`

---

## 6. 推荐的最小可实施组合

第一版就正式支持下面 4 种：

1. `value_type = number` + `rule_type = number_range`
2. `value_type = boolean` + `rule_type = boolean_equal`
3. `value_type = enum` + `rule_type = enum`
4. `value_type = text` + `rule_type = text_pattern`

这 4 种基本能覆盖巡检里绝大多数录入场景。

---

## 7. 建议的动态渲染规则

前端建议按下表处理：

| value_type | rule_type | 前端控件 | 说明 |
|---|---|---|---|
| number | number_range | 数字输入框 | 显示单位、范围、精度 |
| boolean | boolean_equal | 按钮组/单选 | 根据 `options` 渲染 |
| enum | enum | 下拉框/单选组 | 根据 `options` 渲染 |
| text | text_pattern | 文本框 | `rule` 为空时不校验，有 `pattern` 时按正则校验 |

---

## 8. Excel 导入建议

Excel 列建议固定如下：

- `2-装置级`
- `3-模块级`
- `4-检测项目`
- `5-检测项明细`
- `数据类型`
- `权重`
- `规则类型`
- `规则`

其中：

- `数据类型` -> 对应数据库 `inspection_items.value_type`
- `规则类型` -> 对应数据库 `inspection_items.rule_type`
- `规则` -> 对应数据库 `inspection_items.threshold`

建议 Excel 中的“规则”直接填写 JSON 字符串。

示例：

```text
number    number_range    {"min": -5, "max": 20, "unit": "℃"}
boolean   boolean_equal   {"options": ["是", "否"], "normal_value": "是"}
enum      enum            {"options": ["纯白", "米白", "乳白"]}
text      text_pattern    {"pattern": "^[A-Z0-9-]+$", "message": "格式不正确"}
text      text_pattern
```

---

## 9. 约束建议

为了避免前后端各自理解不一致，建议增加以下约束：

1. `rule_type` 必须和 `value_type` 匹配，不能乱组合。
2. 除 `text_pattern` 可为空外，`rule` 必须是合法 JSON。
3. `number_range` 下必须保证 `min <= max`。
4. `enum` / `boolean_equal` 下如果有 `options`，则 `normal_value` 或 `normal_values` 必须属于 `options`。
5. `boolean_equal` 建议统一使用对象格式，不建议只写裸数组。
6. `text_pattern` 若为空，则表示普通文本输入，不做格式校验。

---

## 10. 推荐结论

建议按下面这套作为第一版正式规范：

- `number + number_range`
- `boolean + boolean_equal`
- `enum + enum`
- `text + text_pattern`

其中：

- `boolean_equal` 推荐格式：

```json
{
	"options": ["是", "否"],
	"normal_value": "是"
}
```

- `enum` 推荐格式：

```json
{
	"options": ["纯白", "米白", "乳白"],
	"normal_values": ["纯白", "米白"]
}
```

- `text_pattern` 推荐格式：

```json
{
	"pattern": "^[A-Z0-9-]+$",
	"message": "格式不正确"
}
```

- `text_pattern` 也允许为空：

```text
空
```

表示：只渲染文本输入框，不做额外格式限制。

如果你确认这版规范可行，下一步就按这版去做：

1. Excel 导入校验
2. 模板保存校验
3. 执行页按 `value_type + rule_type + rule` 真正动态渲染控件
